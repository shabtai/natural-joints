How we check the target

Six ways to know the target is right.

A linter checks syntax; a schema test catches a null. We check whether the numbers are actually correct — against the intent, the code, the data, and the domain's priors.

1
Natural-Joints strength

Does it match the intent?

We read the spec, tickets, and analysis plan — then check the output actually does what it was meant to, not just that the code runs without error.

e.g. the analysis plan says "primary-analysis records only"; nothing in the code enforces it.
2
Natural-Joints strength

Where the code meets the data

The bugs that only appear when you understand both the code and what the data actually means — cohorts, confounders, batch effects, units, labels. This is what Natural-Joints is built for.

e.g. disease A needs a cohort filter the code never applies — its active flag lives in a column the transform ignored.
3
Natural-Joints strength

Is the statistic sound?

We check the claim against the data's real structure: are these truly independent samples, and is the variability biological — or driven by batch, repeated measurements, or the wrong unit of analysis?

e.g. a result over "thousands of cells" that all come from one donor — an n of 1, not thousands.
4
Natural-Joints strength

Compare the target to priors

Hold each result up against what the domain expects — reference values, known distributions, plausible ranges. A number that looks fine alone but contradicts the prior gets flagged.

e.g. a computed rate that's plausible on its own but far off the known base rate for that cohort.
5
Natural-Joints strength

Catch corrupted source data

Sometimes the input is already wrong. Deep understanding lets us see when the same thing is recorded two ways, or when a value simply can't be real.

e.g. one disease under two names so a join silently splits it — a patient who gains 20 kg in a day, or a male result in a women's-only panel.
6
conventional debugging

Verify the code itself

Classic code bugs that make the transform do the wrong thing — independent of what the data means. The debugging a careful engineer already does.

e.g. a missing loop that processes only the first batch instead of all of them.
And our priors keep growing. Natural-Joints continuously learns biological priors and biological-data standards — reference ranges, ontologies, naming conventions, plausible distributions — so every audit is sharper than the last.
A real catch

85% of the records feeding a primary clinical endpoint were wrong.

Production-grade code from a public FDA submission pilot. Join, parameter, visit window, sort — all correct. No NaNs, no type errors.

data = (
    adlbc.join(adsl_eff, on="USUBJID", how="inner")
    .filter((pl.col("PARAMCD") == "GLUC") & (pl.col("AVISITN") <= 24))
    .sort(["USUBJID", "AVISITN"])
)

Only 309 of the table's 2,054 glucose records belong to the primary analysis. The rest are unscheduled visits, repeat draws, and duplicates — separated by one column, ANL01FL (the CDISC ADaM analysis flag), lost among 298 others. The code never filters on it.

85% of the records feeding the endpoint are wrong. A semantic-contract bug no unit test catches — Natural-Joints found it autonomously.

The trust model

Zero risk — by construction.

Everything it reads is read-only; the only thing it writes is the report. To verify a transformation it must see all of it — code, data on both ends, and the intent behind it — and every one of those is just a read.

Context & intent

Context docs, Slack, tickets — wherever the domain knowledge lives. Read-only, to learn what the transform was meant to do.

GRANT: read-only context
Nothing is posted back.
Run it as paranoid as you like. Drop it on an isolated — even throwaway — machine: data mounted read-only, no write credentials, and only outbound HTTPS to your model API (plus the open web for public-standard checks; deny even that and it still runs). With no write path and no inbound ports, it physically cannot alter, delete, or exfiltrate your data — the worst case is a wasted run. When it finishes, copy out the report and wipe the box. The only secret it ever needs is your model API key.
Get started

From install to audit report.

Simplest path: open Claude Code where your data and code live and ask it to "install Natural-Joints and audit this pipeline" — it fetches the engine over read-only git, picks the right extras, and runs each step for your approval. Or do it yourself in five lines:

pip install "semantic-enricher[clinical]"                # install · or [gcs] [iceberg] [r] [sas]
export ANTHROPIC_API_KEY=sk-ant-...                     # the only secret it ever needs
semantic-enricher enrich gs://my-bucket/source/ -o ./audit-out # 1 · understand the data
/audit transform.py::build_cohort -d ./audit-out           # 2 · audit the code (in Claude Code)
semantic-enricher view ./audit-out                         # 3 · read the report · localhost:8458

Least privilege throughout: a viewer role on the source, write only to a separate output, outbound HTTPS only — no inbound, no write-back.

What it does — and what it can't

The safety story isn't a promise in a doc; it's what the permissions physically allow.

Natural-Joints does

  • Read source, target, code, and context — all read-only
  • Build deep, domain-aware understanding of every column
  • Verify each step of the transformation against it
  • Write findings & reports to an output dir you name
  • Run entirely from code you can read and pin

Natural-Joints never

  • Writes, updates, or deletes a source row
  • Needs more than a viewer role on your data
  • Opens an inbound port or exposes your data to the internet
  • Requires write access to your code repository
  • Sends data anywhere except your chosen model API
